## 🎯 Resumen Ejecutivo

### De la Especificación a la Realidad

Esta Pull Request cierra el **tercer y último hito del proyecto Adresles**: la transformación del Monorepo Mock (Etapa 2) en un **MVP completamente funcional desplegado en producción**.

**¿Qué implica?**

- **Inteligencia Artificial real**: El Worker conversa con los clientes mediante OpenAI `gpt-4o-mini`, interpreta intenciones, extrae direcciones y guía 9 fases de un journey conversacional — todo en producción, sin mocks.
- **Abstracción LLM**: Interfaz `ILLMService` con implementaciones `OpenAILLMService` y `MockLLMService`, seleccionada en runtime según la presencia de `OPENAI_API_KEY`. Desacopla el motor IA del procesador de conversaciones.
- **Infraestructura de producción completa**: Docker multi-stage builds → AWS ECR → Lightsail ($12/mes) + Caddy HTTPS automático + Vercel (frontend). CI/CD con GitHub Actions para despliegue continuo.
- **DynamoDB en AWS real**: Migración de DynamoDB Local a tablas en AWS con entornos separados (`dev` en eu-west-1, `prod` en eu-central-1) e IAM de mínimo privilegio.
- **Dashboard Admin mejorado**: Página de direcciones, renderizado Markdown en el chat, smart defaults en el simulador, consolidación de `externalOrderId` como referencia única.

**Valor estratégico**:

Esta PR demuestra que Adresles funciona como producto real. Un usuario puede acceder a `https://simulator.adresles.com`, iniciar una simulación de compra, y mantener una conversación en tiempo real con un agente IA que recopila y valida su dirección de entrega. El backend en `https://backend.adresles.com` orquesta todo el flujo: API → BullMQ → Worker (OpenAI + Google Maps) → Redis Pub/Sub → SSE → Frontend.

## 🚀 ¿Qué Cambia y Por Qué?

### 1️⃣ Motor de IA Real — Abstracción LLM

**¿Qué es?**

En la Etapa 2, el Worker devolvía respuestas estáticas. Ahora implementa una capa de abstracción (`ILLMService`) con dos implementaciones intercambiables:

- **`OpenAILLMService`**: Llama a `gpt-4o-mini` con tres métodos especializados: `generateMessage` (conversación), `extractAddress` (parsing de direcciones), `interpretIntent` (detección de intención del usuario).
- **`MockLLMService`**: Consolida los mocks dispersos en una única clase sin dependencias de red, usada cuando `OPENAI_API_KEY` no está definida.

La selección ocurre en `main.ts` del Worker: si hay API key, se instancia OpenAI; si no, se inyecta el mock vía `setLLMService()`.

**¿Por qué?**

- ✅ **Desacoplamiento**: `conversation.processor.ts` y `address.service.ts` no importan `openai` directamente
- ✅ **Testabilidad**: 41 tests pasan sin necesidad de API key ni red
- ✅ **Extensibilidad**: Cambiar de proveedor LLM (Claude, Gemini) solo requiere una nueva implementación de `ILLMService`
- ✅ **Coste controlado**: `gpt-4o-mini` en lugar de `gpt-4`, reducción significativa de coste por conversación

**Impacto**: El Worker pasa de ser un simulador a ser un agente conversacional real capaz de entender lenguaje natural, extraer direcciones estructuradas y guiar al usuario a través de 9 fases de checkout.

---

### 2️⃣ Infraestructura de Producción (Docker + ECR + Lightsail + Caddy)

**¿Qué es?**

Stack completo de despliegue en servidor único que ejecuta el MVP por ~$12/mes:

| Componente         | Tecnología                                    | Propósito                              |
| ------------------ | --------------------------------------------- | -------------------------------------- |
| **Compute**        | AWS Lightsail `small_3_0` (2 GB RAM)          | Servidor Ubuntu con IPv4 pública       |
| **Registry**       | AWS ECR (`eu-central-1`)                      | Imágenes Docker privadas               |
| **Reverse Proxy**  | Caddy 2 (`caddy:2-alpine`)                    | HTTPS automático (Let's Encrypt)       |
| **Backend**        | Docker container (NestJS)                     | `backend.adresles.com`                 |
| **Worker**         | Docker container (BullMQ + OpenAI)            | Procesamiento asíncrono de IA          |
| **Redis**          | Docker container (`redis:7-alpine`)           | Colas + Pub/Sub                        |
| **Frontend**       | Vercel (Free tier)                            | `simulator.adresles.com`               |
| **CI/CD**          | GitHub Actions                                | Push a `main` → ECR → SSH → Compose   |

**Dockerfiles multi-stage** para API y Worker: builder (monorepo completo + compilación TypeScript) → runner (Alpine mínimo con solo `dist/` y `node_modules` de producción).

**¿Por qué?**

- ✅ **Coste mínimo**: Stack completo por ~$12/mes (Lightsail) + $0 (Vercel free tier)
- ✅ **HTTPS sin configuración**: Caddy obtiene certificados Let's Encrypt automáticamente
- ✅ **Reproducibilidad**: Docker Compose idéntico para desarrollo y producción
- ✅ **Aislamiento DNS**: `adresles.com` (WordPress) no se toca; backend usa subdominio `backend.adresles.com`
- ✅ **CI/CD completo**: Push a `main` → build → ECR → SSH → Compose up en ~3 minutos

**Impacto**: Adresles pasa de ejecutarse exclusivamente en local a ser un producto accesible desde Internet con HTTPS, CI/CD y despliegue automatizado.

---

### 3️⃣ DynamoDB en AWS — Entornos Separados

**¿Qué es?**

Migración de DynamoDB Local (Docker) a **dos tablas reales en AWS**, una por entorno:

- `adresles-messages-dev` en `eu-west-1` (Irlanda) → IAM User `adresles-app-dev`
- `adresles-messages-prod` en `eu-central-1` (Frankfurt) → IAM User `adresles-app-prod`

Cada IAM User tiene una política de mínimo privilegio: solo `PutItem`, `GetItem`, `Query`, `UpdateItem` sobre su tabla específica.

**¿Por qué?**

- ✅ **Persistencia real**: Los mensajes sobreviven reinicios y despliegues
- ✅ **Aislamiento de seguridad**: Las credenciales de Dev no pueden operar sobre la tabla de Prod
- ✅ **Coste cero en MVP**: `PAY_PER_REQUEST` dentro del free tier de DynamoDB
- ✅ **Cambio de código mínimo**: Solo se parametrizó `TABLE_NAME` con variable de entorno
- ✅ **Retrocompatible**: DynamoDB Local sigue funcionando sin cambios

**Impacto**: Las conversaciones de producción persisten en AWS con TTL automático de 90 días y seguridad IAM real.

### 4️⃣ Mejoras en el Worker — Conversaciones Enriquecidas

**¿Qué incluye?**

- **Formateo Markdown**: El Worker genera respuestas con formato estructurado (listas, negritas, separadores) que el frontend renderiza como HTML rico.
- **Single-writer pattern**: Los mensajes del usuario se persisten exclusivamente en el Worker (no en la API), eliminando duplicados en DynamoDB.
- **Mejoras en journey INFORMATION**: Mensajes de confirmación de pedido con formato visual mejorado (resumen de productos, totales, estado).

**¿Por qué?**

- ✅ **Experiencia de usuario**: Respuestas legibles con formato visual en lugar de texto plano
- ✅ **Integridad de datos**: Un solo punto de escritura elimina duplicados e inconsistencias
- ✅ **Claridad conversacional**: El usuario ve información del pedido con formato de recibo

---

### 5️⃣ Dashboard Admin — Funcionalidades Expandidas

**¿Qué incluye?**

- **Página de Direcciones**: Nueva sección `/addresses` con tabla completa (ordenación, filtros por texto y favoritos, chips de filtros activos), siguiendo los mismos patrones de `/orders` y `/users`.
- **Renderizado Markdown en chat**: Componente `MarkdownMessage` que transforma el Markdown del asistente IA en HTML accesible con estilos TailwindCSS.
- **Smart defaults en simulador**: El modal de configuración preselecciona la primera tienda y usuario disponibles, y filtra usuarios por la tienda seleccionada.
- **Consolidación de `externalOrderId`**: El ID externo pasa a ser la referencia única para órdenes en UI, búsquedas y ordenación. `ExternalOrderIdService` genera IDs deterministas cuando no vienen del eCommerce.

**¿Por qué?**

- ✅ **Completitud del MVP**: Todas las entidades del dominio (pedidos, usuarios, direcciones) son navegables
- ✅ **Calidad de producto**: El chat muestra contenido formateado, no texto crudo
- ✅ **Usabilidad del simulador**: Menos clics para iniciar una simulación (preselección inteligente)

### 6️⃣ Evolución de Estándares y Workflow

**¿Qué incluye?**

- **Migración de reglas**: Los archivos `*-standards.mdc` se mueven de `openspec/specs/` a `.cursor/rules/`, quedando accesibles como reglas activas del IDE.
- **Workflow multi-agente OpenSpec**: Nuevo archivo `openspec-agent-workflow.mdc` que define las 6 fases del desarrollo y los 4 agentes especializados (architect, backend, frontend, QA).
- **Task tagging**: Regla `openspec-task-tagging.mdc` para etiquetar cada tarea en `tasks.md` con `[architect]`, `[backend]`, `[frontend]` o `[qa]`.

**¿Por qué?**

- ✅ **Productividad**: Las reglas activas del IDE guían a la IA automáticamente, sin necesidad de referenciarlas manualmente
- ✅ **Estructura de equipo**: Incluso en desarrollo individual, la separación de roles garantiza que cada cambio se diseña, implementa y valida con perspectivas diferentes

## 🎯 Decisiones Arquitecturales (ADRs)

### Nuevos ADRs (010-011)

#### ADR-010: DynamoDB en AWS con Entornos Separados e IAM de Mínimo Privilegio

- ✅ **Decisión**: Dos tablas DynamoDB en AWS real (`adresles-messages-dev`, `adresles-messages-prod`), cada una con un IAM User dedicado y política restringida al ARN de su tabla
- 🎯 **Razón**: Aislamiento total Dev/Prod; coste cero con `PAY_PER_REQUEST`; retrocompatibilidad con DynamoDB Local
- 📦 **Recursos**: 2 tablas, 2 IAM Users, 2 políticas IAM, script de validación `validate-dynamodb-aws.ts`
- 🔗 **Complementa**: ADR-002 (decisión original de usar DynamoDB para mensajes)

#### ADR-011: Despliegue en Producción — Docker + ECR + AWS Lightsail + Caddy

- ✅ **Decisión**: Servidor único Lightsail ($12/mes) con Docker Compose (API + Worker + Redis + Caddy), imágenes en ECR, CI/CD con GitHub Actions
- 🎯 **Razón**: Coste mínimo para MVP; HTTPS automático sin configuración; aislamiento del dominio WordPress
- 📡 **URLs**: `backend.adresles.com` (API), `simulator.adresles.com` (Dashboard)
- 🔗 **Reemplaza**: Plan original de Konsole H + Traefik (nunca implementado)
