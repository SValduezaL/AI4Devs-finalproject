## 🎯 Resumen Ejecutivo

### De la Especificación a la Realidad

Esta Pull Request materializa el **segundo hito del proyecto Adresles**: la transformación de la planificación documental (Etapa 1) en un **Monorepo funcional** capaz de ejecutar el flujo completo del MVP en modo Mock.

**¿Qué implica?**

- **Esqueleto Monorepo operativo**: Turborepo orquesta `apps/` (api, web-admin, worker) y `packages/` (prisma-db, shared-types) con un grafo de dependencias bien definido.
- **Flujo funcional end-to-end**: Un pedido mock se crea en la API, se encola en BullMQ, el Worker lo procesa con datos estáticos y el frontend recibe eventos en tiempo real via SSE + Redis Pub/Sub.
- **Validación sin integraciones reales**: OpenAI y Google Maps están preparados en la arquitectura pero devuelven datos estáticos, permitiendo validar UX, flujos conversacionales y sincronización de estado sin coste de APIs externas.

**Valor estratégico**:

Esta PR demuestra que la arquitectura documentada en los ADRs **funciona**. El paso a Monorepo con separación de preocupaciones (`api` → orquestación, `worker` → procesamiento IA, `web-admin` → interfaz) está validado. El siguiente hito será sustituir los mocks por integraciones reales.

## 🚀 ¿Qué Cambia y Por Qué?

### 1️⃣ Evolución a Monorepo (Turborepo)

**¿Qué es?**

Estructura de workspace con `apps/` para aplicaciones ejecutables y `packages/` para código compartido. Turborepo gestiona el grafo de dependencias (`dependsOn: ["^build"]`) y el caché de builds.

**Estructura clave**:

```
apps/
├── api/        # NestJS — orquestación, endpoints, encolado BullMQ
├── web-admin/  # Next.js 14 — dashboard de administración y simulador
└── worker/    # Node.js — procesamiento asíncrono de conversaciones IA

packages/
├── prisma-db/     # @adresles/prisma-db — schema Prisma, migraciones, cliente compartido
└── shared-types/  # @adresles/shared-types — tipos de contratos BullMQ, DTOs compartidos
```

**¿Por qué?**

- ✅ **Single source of truth** para schema (Prisma) y tipos de cola (Shared Types)
- ✅ **Builds incrementales**: Solo se reconstruye lo afectado
- ✅ **Orden correcto**: `dev` depende de `^build` — evita race conditions al arrancar
- ✅ **Escalabilidad**: Nuevas apps (p.ej. plugin eCommerce) se añaden sin tocar el core

**Impacto**: Elimina drift entre API y Worker en tipos y schema. Las migraciones viven en `packages/prisma-db`; API y Worker importan desde `@adresles/prisma-db`.

---

### 2️⃣ Backend & Eventos en Tiempo Real

**¿Qué incluye?**

- **Módulo Mock**: `POST /api/mock/orders` crea pedidos; `POST /api/mock/conversations/:id/reply` permite respuestas simuladas. Todo el flujo conversacional se ejecuta sin llamadas reales a OpenAI.
- **Infraestructura SSE/Redis**: El Worker publica eventos en Redis Pub/Sub (`conversation:{id}:update`). El API suscribe y reenvía via SSE. El frontend consume con `EventSource`.

**¿Por qué?**

- ✅ **API no bloqueada**: Responde `201 Created` en < 200ms; el Worker procesa en background
- ✅ **Tiempo real sin WebSockets**: SSE unidireccional es suficiente (Worker → Frontend)
- ✅ **Reutilización de Redis**: BullMQ ya estaba; Pub/Sub no añade infraestructura nueva

**Impacto**: El simulador de chat en `/simulate` muestra mensajes del asistente en tiempo real mientras el Worker procesa la conversación.

---

### 3️⃣ Frontend & Simulación

**¿Qué incluye?**

- **Dashboard Admin**: Páginas de Orders (ordenación, filtros, búsqueda), Users (registro, búsqueda) y navegación lateral.
- **Simulador de Chat**: Configuración de pedido (usuario, tienda, modo), chat en vivo con SSE, indicador de escritura, badges de estado de conversación.
- **Implementación de Sistema de Diseño base (Shadcn/UI)**: Componentes reutilizables (Button, Dialog, Table, Select, Badge, etc.) para consistencia visual.

**¿Por qué?**

- ✅ **Validación UX**: Se puede probar el flujo conversacional completo sin coste de APIs
- ✅ **Iteración rápida**: Cambios en mensajes o flujos no requieren despliegue de IA
- ✅ **Base para producción**: El layout y patrones se reutilizarán en la app conversacional real

**Impacto**: Permite demos y QA del caso de uso "Procesar Compra" antes de la integración con OpenAI y Google Maps.

## 🎯 Decisiones Arquitecturales (ADRs 005-009)

Esta PR incorpora **5 ADRs nuevos** que definen la arquitectura operativa del MVP Mock:

### ADR-005: BullMQ + Worker Dedicado para Conversaciones IA

- ✅ **Decisión**: API encola jobs en Redis; Worker independiente los consume de forma asíncrona
- 🎯 **Razón**: Latencia de IA (1–5 s) inaceptable en síncronía; reintentos automáticos; escalado independiente
- 📦 **Colas**: `process-conversation` (inicio), `process-response` (por turno de usuario)

### ADR-006: SSE + Redis Pub/Sub para Real-Time

- ✅ **Decisión**: Worker publica en Redis; API suscribe y reenvía al cliente via SSE
- 🎯 **Razón**: Unidireccional (Worker → Frontend); Redis ya disponible; sin polling
- 📡 **Canales**: `conversation:{conversationId}:update` para mensajes y eventos terminales

### ADR-007: Paquete `packages/shared-types`

- ✅ **Decisión**: Tipos compartidos (BullMQ, DTOs) centralizados en `@adresles/shared-types`
- 🎯 **Razón**: Evitar drift entre API y Worker; garantía en compilación TypeScript
- 📦 **Exporta**: `ProcessConversationJobData`, `ProcessResponseJobData`, `MockOrderContext`

### ADR-008: Worker apunta al schema Prisma del API (Opción C — stepping stone)

- ✅ **Decisión**: Worker elimina su schema; lee `../api/prisma/schema.prisma` vía `package.json`
- 🎯 **Razón**: Corrección urgente del drift; solución temporal hasta ADR-009
- ⚠️ **Obsoleto**: Sustituido por ADR-009 en esta PR

### ADR-009: Paquete `packages/prisma-db` — Fuente Única del Schema

- ✅ **Decisión**: `@adresles/prisma-db` contiene schema, migraciones, seed y cliente generado
- 🎯 **Razón**: Dependencia explícita; Turborepo gestiona el grafo; patrón coherente con ADR-007
- 📦 **Responsabilidad**: API y Worker importan `PrismaClient` desde `@adresles/prisma-db`
